# zmol v0.1 — implementation plan

The single goal of v0.1: a working desktop binary that loads a PDB structure and accepts both PyMOL and ChimeraX selection grammars, rendering on Mol\*.

If v0.1 ships and does that one thing, the unified-grammar idea is proven, the funding applications have a concrete artifact to point at, and v0.2 onward becomes a long tail of command coverage.

## Scope (do)

- Desktop binary for one platform (Linux to start; macOS / Windows in v1.0)
- Mol\* WebGL viewer rendering inside an Electron window
- A command-line console pinned to the bottom of the window
- Load a PDB by ID: `load 1cbs`
- Load a CIF / PDB from local file: `load /path/to/file.pdb`
- Both selection grammars in parallel:
  - PyMOL: `select chain A and resi 50-60 and name CA`
  - ChimeraX: `select /A:50-60@CA`
  - The console accepts either; the parser figures out which.
- Apply a representation change to a selection: `show sticks, sele` (PyMOL) or `style sel stick` (ChimeraX)
- Apply a color change: `color red, sele` or `color sel red`
- Reset / clear: `delete all` (PyMOL), `close` (ChimeraX)
- Help: `help` lists the implemented commands
- Settings file (`~/.zmol`) — defines startup defaults

## Scope (do NOT — defer to v0.2+)

- HTTP/WebSocket remote-control API (v0.4)
- Narrated-show generator (v0.5)
- Cross-platform signed binaries (v1.0)
- The full ~30 commands per grammar (v0.2 / v0.3)
- Auto-update mechanism (v1.0)
- Plugin / extension API (v1.x)
- Trajectory / ensemble support (out of grant scope)

The v0.1 milestone is *intentionally* small. Resist scope creep — v0.2 / v0.3 are explicitly the long-tail-of-commands work.

### Three core modules

1. **Grammar router** (`src/grammar/router.ts`)
   - Takes a raw command string
   - Decides PyMOL or ChimeraX based on syntax markers
   - Routes to the right parser

2. **Parsers** (`src/grammar/pymol/parser.ts`, `src/grammar/chimerax/parser.ts`)
   - Each parser produces a unified intermediate representation (IR)
   - Same IR for both grammars — that's the whole point

3. **IR → Mol\* compiler** (`src/molstar-compiler/`)
   - Takes IR
   - Produces Mol\* `StateBuilder` transactions
   - Applies them to the viewer instance

### The unified intermediate representation (IR)

```typescript
type Selection = {
  type: 'all' | 'chain' | 'residue' | 'atom' | 'and' | 'or' | 'not'
  // ...payload depends on type
}

type Command =
  | { kind: 'load', source: 'pdbid' | 'file', value: string }
  | { kind: 'select', name: string, selection: Selection }
  | { kind: 'show', representation: Representation, selection: Selection }
  | { kind: 'hide', representation: Representation, selection: Selection }
  | { kind: 'color', color: Color, selection: Selection }
  | { kind: 'delete', selection: Selection }
  | { kind: 'close' }
  | { kind: 'help', topic?: string }

type Representation = 'cartoon' | 'sticks' | 'spheres' | 'surface' | 'lines'
type Color = string  // CSS color or named
```

Both PyMOL `show sticks, sele` and ChimeraX `style sel stick` lower to:
```
{ kind: 'show', representation: 'sticks', selection: <ref to sele> }
```

### Grammar detection heuristics

PyMOL distinctive markers:
- Comma-separated arguments: `color red, chain A`
- Word operators: `and`, `or`, `not`
- Selection by `name`, `resi`, `chain` keywords
- `name CA` (atom-name selection)

ChimeraX distinctive markers:
- Slash-prefixed chain ranges: `/A:50-60`
- `@` for atom names: `@CA`
- Colon separators: `:LYS`, `:50-60`
- Space-delimited arguments: `style sel stick`

Heuristic (v0.1):
1. If the command contains `,` outside of quotes, try PyMOL first
2. If the command contains `/`, `:`, or `@` as selection delimiters, try ChimeraX first
3. If both parsers reject, return a parse error

This is good-enough for v0.1. Later versions will allow explicit `pymol> ...` / `chimerax> ...` prefixes for ambiguous cases.

## Tech stack (proposed)

| Layer | Choice | Why |
|---|---|---|
| Desktop shell | Electron (latest stable) | Smol6 used it; works; ships everywhere |
| Renderer | TypeScript + Mol\* | Mol\* is the rendering core by design |
| Parser | hand-written recursive descent (TypeScript) | parser-generators are overkill for these grammars; hand-written is more debuggable |
| Test framework | Vitest | Fast, TypeScript-native |
| Linter / formatter | Biome | Single tool replaces ESLint + Prettier |
| Build | esbuild via electron-vite | Fast bundling, hot reload |
| Packaging | electron-builder | AppImage for Linux v0.1 |

## Suggested implementation order (10–15 working days for v0.1)

### Week 1 — foundation

**Day 1–2:** scaffold the Electron + TypeScript + Mol\* project. Get a window with a Mol\* viewer that loads `1cbs` from a hard-coded button. No console yet.

**Day 3:** add the console UI (input, history, scroll). Hook `Enter` to log the command verbatim — no parsing yet.

**Day 4:** implement the simplest command — `load <pdbid>`. Direct Mol\* state-tree call, no parser yet. Confirm a typed `load 1cbs` loads the structure.

**Day 5:** define the IR types. Write the grammar router skeleton — for now it returns `{kind: 'unknown'}` for everything. Wire it through.

### Week 2 — the unified grammar

**Day 6:** implement the PyMOL parser for `select`, `show`, `color`. Hand-written recursive descent. Tests for each.

**Day 7:** implement the ChimeraX parser for the same three commands. Same pattern.

**Day 8:** implement the IR → Mol\* compiler for `select`, `show`, `color`. The hardest part: Mol\*'s state-tree wants you to declare the whole structure transformation up front, which is at odds with imperative console commands.

**Day 9:** implement `delete`, `close`, `help`. Add the grammar-detection heuristic to the router.

**Day 10:** write 30+ test cases of PyMOL and ChimeraX selections that should produce the same IR. End-to-end test: typed command → IR → Mol\* state change.

### Week 3 — polish + ship

**Day 11–12:** stabilize. Fix the inevitable bugs. Add settings file loading (`~/.zmol`).

**Day 13:** docs — `README.md` update with v0.1 features, `COMMAND-COVERAGE.md` listing what works, demo gif.

**Day 14:** package as Linux AppImage. Sanity-test on a clean Ubuntu VM.

**Day 15:** release `v0.1` on GitHub. Show HN post. Medium launch post.

If any week slips by 2+ days, cut scope (e.g., drop `color`, ship with only `select` + `show`). Shipping a smaller v0.1 on time beats a feature-complete v0.1 a month late.

## Test corpus

The single most important artifact in v0.1 is the test corpus: 30+ pairs of equivalent PyMOL and ChimeraX commands that should produce identical IR. This corpus is the proof that the unified grammar idea works.

Examples to start with:

| PyMOL | ChimeraX | Should produce |
|---|---|---|
| `select sel, chain A` | `select /A` | `Selection.chain('A')` |
| `select sel, resi 50-60` | `select :50-60` | `Selection.residue_range(50, 60)` |
| `select sel, chain A and resi 50-60` | `select /A:50-60` | `Selection.and(chain('A'), residue_range(50,60))` |
| `select sel, name CA` | `select @CA` | `Selection.atom_name('CA')` |
| `select sel, chain A and resi 50-60 and name CA` | `select /A:50-60@CA` | full chained selection |
| `show sticks, sel` | `style sel stick` | `Show(sticks, ref('sel'))` |
| `color red, sel` | `color sel red` | `Color(red, ref('sel'))` |
| `delete all` | `close` | `Delete.all` |

Commit this corpus as `tests/grammar/unified.test.ts`. When a reviewer / funder asks "does the unified grammar idea actually work," point at the test file.

## Risk register

| Risk | Likelihood | Mitigation |
|---|---|---|
| Mol\* state-tree is too declarative for imperative console | High | Use the imperative-mode helpers in `mol-plugin` rather than the React-style `StateBuilder` directly |
| Selection-language edge cases (PyMOL has surprising semantics) | High | Cover the obvious cases in v0.1; flag remaining as `parse error: not yet supported` rather than silently wrong |
| Electron packaging produces broken AppImages | Medium | electron-builder is mature; test on a clean VM |
| Performance regressions from running both parsers | Low | Parsers are cheap; parsing 1KB of input takes microseconds |
| smol6 has architectural lessons I haven't extracted | High | **Read smol6's command dispatcher before writing zmol's** — the lessons are there |

## Lessons to pull from smol6

Before starting v0.1 implementation, do a lessons-learned pass on `~/github/sness23/smol6/`:

- What command-dispatch structure did smol6 land on? Adopt or improve.
- What were smol6's biggest pain points? Avoid.
- What worked well? Keep.
- What dependencies did smol6 use? Use the same versions in zmol unless there's a reason to change.

Track lessons in `docs/SMOL6-LESSONS.md` once you've reviewed.

## What "the funding application is unblocked" looks like

The NLnet / EV applications can be submitted before v0.1 ships. But once v0.1 ships:

- The "honest status: pre-v0.1" disclaimers in the README disappear
- The Show HN post can go out, generating early adoption signals
- The arXiv preprint can include a "tools available" line linking zmol
- CZI EOSS Cycle 8 application becomes credible — by Cycle 8 you'll have v0.1 → v0.5 of usage data

Aim to have v0.1 shipped within 4 weeks of an NLnet acceptance. If NLnet doesn't accept and EV does, the timeline is the same — EV's $25k–40k buys you the same 4 weeks of focused work.
